# Configuring Rudder with `rudder.yml`

Rudder is **configured from code**. Commit a `rudder.yml` file to the root of your
connected Git repository and the control-plane reads it on every reconcile (each
Git pull). Change a value, commit, and it takes effect on the next reconcile — no
restart, no clicking around a UI.

`rudder.yml` has three **optional** blocks:

| Block | What it controls |
|---|---|
| [`settings:`](#settings--operational-tuning) | Operational tuning — reconcile interval, run limits/timeout, SSH policy, host reachability ("keep-alive") |
| [`alerts:`](#alerts--notifications) | Notification channels (Slack / Teams / webhook / log) and when they fire |
| [`dashboard:`](#dashboard--overview-layout) | The Overview dashboard layout (usually generated by the UI's **Copy dashboard YAML**) |

Every block is optional — omit one and Rudder uses its built-in defaults. You can
start with an empty repo and add only what you want to change.

> **🔒 Never put secrets in `rudder.yml`.** It lives in Git. API keys, Git tokens,
> SSH private keys, and vault passwords are provided out-of-band (a Kubernetes
> Secret / Vault) and are never read from this file. `rudder.yml` holds only
> non-secret operational settings.

How values are handled:
- **Defaults always apply.** Anything you don't set keeps its default (shown below).
- **Invalid values are ignored, not fatal.** Out-of-range numbers are clamped to
  the allowed range; junk (e.g. a string where a number is expected) is ignored and
  the default is kept. A bad commit can never take down reconcile.
- **Removing a block reverts to defaults** on the next reconcile.

## `settings:` — operational tuning

```yaml
settings:
  reconcileSeconds: 120          # how often Rudder pulls Git + re-renders the schedule
  runWorkers: 4                  # how many playbook runs may execute at once
  runQueueMax: 20                # max runs in flight (running + queued) before /run returns 429
  runTimeoutSeconds: 1800        # hard kill for a single run (0 = no timeout)
  sshStrict: false               # require hosts in known_hosts (true) vs trust-on-first-use (false)

  reachability:                  # the Inventory "is this host up?" probe ("keep-alive")
    intervalSeconds: 60          # how often every host is probed
    timeoutSeconds: 3            # per TCP connect attempt
    attempts: 2                  # connect attempts per probe before it counts as a failure
    downAfter: 3                 # consecutive failed probes before a host is marked disconnected
```

### Reference

| Key | Default | Range | What it does |
|---|---|---|---|
| `reconcileSeconds` | `120` | 10–86400 | Seconds between Git pulls. Lower = picks up commits faster; higher = less Git traffic. |
| `runWorkers` | `4` | 1–64 | Concurrent playbook runs. Raise to run more jobs in parallel (uses more CPU/RAM). Applied when the run pool is idle. |
| `runQueueMax` | `20` | 1–1000 | Total runs allowed in flight (running + queued). Beyond this, `POST /run` returns HTTP 429 (backpressure). |
| `runTimeoutSeconds` | `1800` | 0–86400 | A run still going after this is force-killed (SIGKILL of the whole process group). `0` disables the timeout. |
| `sshStrict` | `false` | bool | `false` = trust a host's SSH key on first contact (rejects later key changes). `true` = the host must already be in `known_hosts` or the run fails. |
| `reachability.intervalSeconds` | `60` | 5–3600 | How often each inventory host is probed for up/down. |
| `reachability.timeoutSeconds` | `3` | 0.5–30 | TCP connect timeout per attempt. Raise for slow/distant hosts. |
| `reachability.attempts` | `2` | 1–5 | Connect attempts within one probe before it's a failure (absorbs a single dropped packet). |
| `reachability.downAfter` | `3` | 1–20 | A host stays **up** until it fails this many probes **in a row**, then shows disconnected. This is the anti-flap control — raise it if hosts blink between active/disconnected. |

> **Tuning the "flapping" of Inventory hosts:** a host is "up" if a TCP connection
> to its SSH/management port succeeds. With the defaults a host must be unreachable
> for ~3 minutes (`downAfter: 3` × `intervalSeconds: 60`) before it shows
> disconnected, and a single success brings it straight back. If your network is
> jittery, raise `downAfter` and/or `timeoutSeconds`.

## `alerts:` — notifications

A list of channels. When a job finishes, Rudder computes an **event** and sends to
every channel whose `on:` list matches.

Events:
- `failed` — the run failed.
- `success` — the run succeeded.
- `recovery` — a run succeeded **after** the previous one failed. (A channel that
  asked for `success` also receives `recovery`.)

```yaml
alerts:
  - type: slack                          # slack | teams | webhook | log
    target: https://hooks.slack.com/services/XXX/YYY/ZZZ
    on: [failed, recovery]               # which events to send

  - type: webhook                        # generic JSON POST
    target: https://example.com/rudder-hook
    on: [failed]
    jobs: [nightly-patching]             # optional: only these jobs (omit = all jobs)

  - type: log                            # write to the control-plane log (no target needed)
    on: [failed, success, recovery]
```

| Field | Required | Notes |
|---|---|---|
| `type` | yes | `slack` / `teams` = post to an incoming-webhook URL; `webhook` = generic JSON POST; `log` = print to the control-plane log. |
| `target` | for slack/teams/webhook | The webhook URL. (A URL is not a secret in the sense that matters here, but if yours embeds a token you consider sensitive, front it with a proxy.) |
| `on` | yes | Any of `failed`, `success`, `recovery`. |
| `jobs` | no | Restrict the channel to specific job names. Omit to apply to all jobs. |

Test a channel from the UI (**Settings → channel → Test**) or `POST /channels/test`.

## `dashboard:` — Overview layout

Controls how the Overview page is arranged. You normally **don't hand-write this** —
edit the dashboard in the UI (**Edit dashboard** → drag/resize/add widgets), then
click **Copy dashboard YAML** and paste the block here. Committing it makes your
layout survive any rebuild (it's read from Git on reconcile).

```yaml
dashboard:
  cols: 12                                         # grid width (default 12)
  widgets:
    - { type: verdict,           x: 0, y: 0, w: 12, h: 3 }
    - { type: metrics-summary,   x: 0, y: 3, w: 12, h: 3 }
    - { type: server-resources,  x: 0, y: 6, w: 5,  h: 6 }
    - { type: metric, metric: success-rate, x: 5, y: 6, w: 4, h: 3 }
```

Each widget is `{ type, x, y, w, h }` (grid units). Generic metric widgets also take
a `metric:` field. Unknown widget types are skipped (a bad entry can't break the
page). Widget types: `verdict`, `metrics-summary`, `needs-attention`,
`fleet-at-a-glance`, `server-resources`, `reconcile`, `connected-repos`,
`upcoming-runs`, and `metric` (with `metric:` one of `success-rate`,
`avg-run-duration`, `group-reachability`, `queue-depth`, `jobs-by-status`,
`next-run`, `recent-activity`). Omit the whole block to use the default layout.

## Full example

```yaml
# rudder.yml — committed to the root of your fleet repo.

settings:
  reconcileSeconds: 120
  runWorkers: 4
  runQueueMax: 20
  runTimeoutSeconds: 1800
  sshStrict: false
  reachability:
    intervalSeconds: 60
    timeoutSeconds: 3
    attempts: 2
    downAfter: 3

alerts:
  - type: slack
    target: https://hooks.slack.com/services/XXX/YYY/ZZZ
    on: [failed, recovery]

# dashboard: is normally generated by the UI's "Copy dashboard YAML".
```

## Seeing the effective settings

The control-plane exposes the merged result (defaults + your overrides):

```bash
curl -s http://<rudder>/api/control-plane/settings | jq
```

Changes apply on the **next reconcile** (within `reconcileSeconds`), or immediately
if you trigger a reconcile from the UI / `POST /reconcile`.
